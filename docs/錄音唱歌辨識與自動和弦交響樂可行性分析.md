# 錄音辨識唱歌音高、節奏、強度與歌詞，並自動判斷和弦／生成交響樂：方法與可行性分析

## 摘要

可以做到，但需要把目標分成兩類：

1. **辨識／評分型**  
   從錄音中分析音高、節奏、強度、歌詞內容，進一步判斷演唱準確度。  
   這類技術可行性高，適合做成 K 歌評分、聲樂練習、旋律轉 MIDI、歌詞對齊等工具。

2. **創作／編曲型**  
   根據人聲旋律與歌詞，自動推論和弦，再生成鋼琴伴奏、弦樂或交響樂編制。  
   這也可行，但不是「唯一正解」問題。同一段旋律可以配出多種合理和弦與風格，因此比較適合作為**可編輯的編曲草稿產生器**。

---

## 一、整體系統架構

```text
錄音輸入
  ↓
降噪 / 正規化 / 人聲分離
  ↓
音高辨識 + 節奏辨識 + 強度分析 + 歌詞辨識
  ↓
音符切分與量化：轉成旋律線 / MIDI / MusicXML
  ↓
調性分析 + 和弦推論 / 和弦生成
  ↓
伴奏生成：鋼琴 / 吉他 / 弦樂 / 管弦樂
  ↓
輸出：MIDI、樂譜、音訊、分軌、和弦表
```

如果輸入是**清唱單人聲**，成功率會高很多。  
如果輸入是**完整混音歌曲**，通常需要先做 source separation，把 vocal、drums、bass、other 等 stem 分開。

---

## 二、各模組方法與可行性

## 1. 音高辨識

### 可行性：高

單人清唱的音高辨識是目前最成熟的部分之一。

常見方法包括：

- 傳統 DSP：YIN、pYIN、autocorrelation、cepstrum
- 深度學習：CREPE、torchcrepe、Basic Pitch
- Audio-to-MIDI：Basic Pitch、MT3 類模型

可輸出的資料形式如下：

```json
{
  "notes": [
    {
      "start": 0.52,
      "end": 1.14,
      "pitch_midi": 64,
      "pitch_name": "E4",
      "cents_deviation": -12,
      "confidence": 0.91
    }
  ]
}
```

### 難點

歌聲有很多非固定音符特徵：

- 滑音
- 顫音
- 哭腔
- 轉音
- 氣音
- 裝飾音

因此不能把每一個瞬間 F0 都直接當作獨立音符，還需要做：

```text
F0 tracking
  → note segmentation
  → pitch quantization
  → MIDI note generation
```

---

## 2. 節奏與拍點辨識

### 可行性：中高

節奏辨識可以拆成幾個任務：

- onset detection：偵測每個音的開始時間
- beat tracking：推測拍點
- tempo estimation：估計 BPM
- meter inference：推測幾拍子，例如 4/4、3/4
- quantization：把自由演唱對齊到節奏格線

推薦流程：

```text
onset detection
  → 初步 tempo / beat grid
  → 依旋律 phrase 修正
  → HMM / dynamic programming / DTW 對齊
  → 輸出可編輯節奏
```

### 難點

歌唱常有：

- rubato，自由速度
- 拖拍
- 搶拍
- 無伴奏清唱時 BPM 不穩
- 前奏或尾音不在固定拍點上

如果是「比對指定樂譜」，節奏評分會相對準確；如果是「完全不知道原曲，只靠清唱推完整節奏」，可行但通常需要人工校正。

---

## 3. 強度／表情辨識

### 可行性：中高

「強度」可以指不同層次：

```text
聲學強度：RMS、LUFS、dB、energy
演唱強度：是否用力、是否爆音、是否氣聲
音樂力度：p、mp、mf、f、ff
情緒張力：溫柔、激昂、壓抑、爆發
```

工程上最容易做的是 note-level loudness：

```json
{
  "note_id": 12,
  "rms_db": -21.4,
  "peak_db": -8.2,
  "relative_velocity": 82,
  "dynamic_mark": "mf"
}
```

### 難點

聲音比較大不一定代表音樂上更「強」。以下因素都會影響 loudness：

- 麥克風距離
- 錄音設備
- 混音壓縮
- 房間殘響
- 背景噪音

如果要做聲樂教學，最好同時分析：

- 音準
- 音色
- 共鳴
- 氣息
- 咬字
- 長音穩定度

---

## 4. 歌詞辨識

### 可行性：中

可以使用：

- Whisper
- wav2vec 2.0 fine-tuning
- Conformer / CTC / attention ASR
- singing-specific ALT 模型

但唱歌歌詞辨識比一般語音辨識困難，原因包括：

- 母音被拉長
- 音高變化大
- 有顫音與轉音
- 咬字不像說話
- 伴奏會干擾
- 訓練資料較少

### 建議策略

如果使用者沒有提供歌詞：

```text
錄音 → ASR / ALT → 猜測歌詞 → 人工校正
```

如果使用者已提供歌詞：

```text
錄音 + 歌詞 → forced alignment → 每個字 / 詞的時間軸
```

已知歌詞的 forced alignment 會比完全從歌聲猜歌詞可靠很多。

---

## 三、自動判斷和弦

這部分要區分兩種情況。

---

## 1. 從完整歌曲辨識既有和弦

### 可行性：中

如果錄音中已經有伴奏，可以透過以下特徵推測和弦：

- chroma
- CQT
- bass line
- harmonic change
- HMM / Viterbi smoothing

輸出範例：

```text
0:00 - 0:02 | C
0:02 - 0:04 | G/B
0:04 - 0:06 | Am
0:06 - 0:08 | F
```

### 難點

完整混音歌曲的和弦辨識會受到以下因素影響：

- 鼓聲與噪音
- 混音方式
- 轉位和弦
- 經過和弦
- 低音線與和弦根音不一致
- 非功能和聲

---

## 2. 從清唱旋律生成和弦

### 可行性：中高

這類任務通常稱為 **melody harmonization**。  
它不是「辨識唯一答案」，而是「產生合理和聲」。

同一段旋律可以配出不同和弦：

```text
C - G - Am - F
C - Em - F - G
Am - F - C - G
Cmaj7 - Dm7 - G7 - Cmaj7
```

比較好的系統應該輸出多組候選，而不是只輸出一組：

```json
{
  "key": "C major",
  "chord_candidates": [
    {
      "style": "pop",
      "progression": ["C", "G", "Am", "F"],
      "confidence": 0.82
    },
    {
      "style": "jazz_ballad",
      "progression": ["Cmaj7", "E7", "Am7", "Dm7 G7"],
      "confidence": 0.67
    },
    {
      "style": "cinematic",
      "progression": ["C", "Gsus4", "Am", "Fadd9"],
      "confidence": 0.74
    }
  ]
}
```

## 四、自動生成交響樂

### 可行性：中

從旋律與和弦變成交響樂，比自動配和弦更難。系統需要決定：

- 主旋律由誰演奏？小提琴、長笛、法國號？
- 和聲由弦樂、木管還是銅管承擔？
- 低音由大提琴、低音提琴、低音號還是鋼琴左手負責？
- 節奏型如何設計？
- 是否需要對位？
- 高潮怎麼鋪陳？
- 歌詞情緒如何轉成配器？

---

## 五、交響樂生成方法

## 方法 1：規則式編曲

範例規則：

```text
主旋律 → Violin I / Flute
副旋律 → Violin II / Oboe
和聲 pad → Viola / Cello
低音 → Contrabass / Tuba
高潮 → Brass + Timpani + Cymbal
```

### 優點

- 可控
- 可解釋
- 容易避免樂器音域錯誤
- 適合輸出可編輯樂譜

### 缺點

- 容易模板化
- 音樂變化較少
- 藝術感需要人工設計

---

## 方法 2：符號式 AI 編曲

流程：

```text
旋律 + 和弦 + 節奏
  → MIDI / MusicXML
  → Transformer / sequence model
  → 多軌 MIDI / 樂譜
```

### 優點

- 可輸出 MIDI 或 MusicXML
- 可編輯
- 適合專業工作流
- 可以結合音樂理論規則

### 缺點

- 訓練資料與版權問題較複雜
- 需要處理聲部合理性
- 若無規則限制，可能產生不可演奏或不自然的聲部

---

## 方法 3：文字／旋律條件式音訊生成

流程：

```text
人聲旋律 + 文字描述
  → 音樂生成模型
  → 類交響樂音訊
```

例如輸入：

```text
根據這段旋律生成壯闊、電影感、弦樂與銅管為主的交響樂伴奏。
```

### 優點

- 生成速度快
- 可以快速得到有氛圍的音訊
- 適合 demo 與概念草稿

### 缺點

- 不一定能輸出乾淨樂譜
- 聲部不可控
- 和弦可能漂移
- 很難逐音修改
- 不適合嚴肅樂譜交付

---

## 六、可行性分級表

| 功能 | 可行性 | 備註 |
|---|---:|---|
| 清唱音高辨識 | 高 | 單人聲效果最好 |
| 清唱轉 MIDI | 高 | Basic Pitch / CREPE 類工具可做 |
| 節奏 / onset 偵測 | 中高 | 自由節奏、rubato 會較難 |
| 音量 / 強度分析 | 中高 | 聲學強度容易，音樂表情較難 |
| 已知歌詞對齊 | 高 | 比「猜歌詞」可靠很多 |
| 未知歌詞辨識 | 中 | 唱歌 ASR 仍比一般語音困難 |
| 完整歌曲人聲分離 | 中高 | 可用 Demucs，但有殘留與 artifact |
| 完整歌曲和弦辨識 | 中 | 受混音、轉位、低音線影響 |
| 清唱自動配和弦 | 中高 | 可產生合理候選，但無唯一正解 |
| 自動鋼琴伴奏 | 中高 | 比交響樂簡單很多 |
| 自動交響樂編曲 | 中 | 可出草稿，專業品質仍需人工修 |
| 即時低延遲演唱分析 | 中 | 需要取捨延遲與準確率 |
| 商用品質全自動交響樂 | 中低 | 音樂審美、聲部控制、資料版權都是瓶頸 |

## 七、建議落地方案

## 第一階段：清唱轉旋律 + 歌詞 + 和弦建議

### 輸入

```text
一段清唱錄音
```

### 輸出

```text
旋律 MIDI
每個音的音高 / 時長 / 強度
歌詞逐字或逐詞時間軸
調性
3～5 組和弦候選
簡單鋼琴伴奏
```

這階段技術風險最低，也最容易做出可展示產品。

---

## 第二階段：加入演唱評分

如果有原曲或目標旋律，可以比較：

```text
音準：每個音偏差幾 cents
節奏：onset / duration 偏差
強度：是否符合原曲力度曲線
歌詞：漏字、錯字、咬字時間
穩定度：顫音、長音抖動、滑音控制
```

適合應用於：

- 聲樂練習
- KTV 評分
- 教學 App
- 兒童音樂訓練
- 練唱輔助工具

---

## 第三階段：加入自動編曲

建議不要一開始就直接做完整交響樂，而是逐步擴充：

```text
melody + chords
  → piano accompaniment
  → guitar / pad accompaniment
  → string quartet
  → chamber orchestra
  → full orchestra
```

較穩定的策略是 **AI + 規則 + 可編輯輸出**：

```text
AI 決定和弦、段落、情緒
規則引擎控制樂器音域與聲部
模型生成伴奏 pattern
最後輸出可編輯 MIDI / MusicXML
```

---

## 八、關鍵技術難點

## 1. 歌聲不是穩定音符

系統要判斷：

```text
這是主音？
這是滑到主音的過渡？
這是顫音？
這是裝飾音？
還是唱不準？
```

如果只是單純追蹤 F0，會產生非常碎的 MIDI。

---

## 2. 歌詞和音符對齊很難

中文、台語、日文、英文的歌唱咬字不同。  
中文還有聲調與旋律的衝突問題；台語／華語混唱、鼻音、連音都會降低 ASR 準確率。

---

## 3. 和弦沒有唯一答案

自動配和弦不是分類題，而是創作題。  
系統應該提供風格化候選，例如：

```text
流行：C - G - Am - F
爵士：Cmaj7 - A7 - Dm7 - G7
電影感：C - G/B - Am - Fadd9
古典：I - V6 - vi - IV
```

---

## 4. 交響樂配器需要音樂知識

模型不能只知道和弦，還要知道：

```text
樂器音域
轉位
voice leading
避免聲部打架
呼吸與換弓
銅管耐力
弦樂 divisi
木管音色混合
打擊樂使用時機
```

這就是為什麼「AI 自動交響樂」最適合先定位成草稿產生器，而不是完全取代編曲家。

## 十、MVP 產品規格建議

## MVP 1：清唱分析器

### 功能

- 上傳清唱錄音
- 偵測旋律音高
- 偵測每個音的起訖時間
- 分析音量曲線
- 轉成 MIDI
- 顯示音準偏差

### 輸出

```text
音高曲線圖
MIDI 檔
每個音的音高、開始時間、結束時間、音量
```

---

## MVP 2：歌詞對齊器

### 功能

- 使用者輸入歌詞
- 系統根據錄音對齊歌詞時間
- 輸出逐字 / 逐詞時間碼

### 輸出

```json
[
  {
    "word": "我",
    "start": 0.42,
    "end": 0.73
  },
  {
    "word": "想",
    "start": 0.74,
    "end": 1.05
  }
]
```

---

## MVP 3：旋律自動配和弦

### 功能

- 根據旋律推測調性
- 生成多組和弦候選
- 支援不同風格

### 風格範例

```text
Pop
Ballad
Jazz
Cinematic
Classical
Anime / J-pop
```

---

## MVP 4：自動伴奏產生器

### 功能

- 根據旋律與和弦產生鋼琴伴奏
- 支援簡單 pattern
- 輸出 MIDI / MusicXML

### 範例 pattern

```text
Block chord
Arpeggio
Broken chord
Left-hand bass + right-hand chord
Pad style
```

---

## MVP 5：管弦樂草稿產生器

### 功能

- 根據旋律、和弦、情緒標籤生成多軌 MIDI
- 支援弦樂、木管、銅管、打擊樂
- 可輸出給 DAW 或 MuseScore 編輯

### 目標

不是一次生成最終商用品質，而是生成：

```text
可聽
可改
可當編曲起點
可輸出樂譜與 MIDI
```

## 十一、工程開發風險

| 風險 | 影響 | 緩解方式 |
|---|---|---|
| 錄音品質差 | 音高與歌詞辨識錯誤 | 加入降噪、音量正規化、錄音指引 |
| 伴奏干擾 | 人聲與和弦辨識不準 | 使用 Demucs 等 source separation |
| 歌聲裝飾音太多 | MIDI 過碎 | 加入 note segmentation 與 smoothing |
| 自由節奏 | 拍點不穩 | 允許人工校正 beat grid |
| 歌詞 ASR 錯誤 | 歌詞時間軸錯誤 | 優先採用已知歌詞 forced alignment |
| 和弦生成主觀 | 使用者不滿意 | 提供多組風格化候選 |
| 交響樂聲部不自然 | 樂譜不可用 | 加入樂器音域與 voice leading 規則 |
| 模型版權 / 訓練資料 | 商用風險 | 使用授權清楚的資料與模型 |

---

## 十二、結論

這個方向**可以做，而且工程上有機會落地**。

最實際的路線不是一開始追求：

```text
唱一段 → 完美交響樂
```

而是分層實作：

```text
1. 唱歌錄音分析：音高、節奏、強度、歌詞
2. 旋律轉 MIDI / 樂譜
3. 自動推調性與和弦候選
4. 產生鋼琴或簡單伴奏
5. 擴充成弦樂 / 管弦樂可編輯草稿
```

真正高品質的系統應採用：

```text
AI 模型
+ 音樂理論規則
+ 可編輯 MIDI / MusicXML 輸出
+ 人工校正介面
```

完全自動產出「可直接商用的交響樂」目前仍不穩定；但自動產出「可聽、可改、可當編曲起點的交響樂草稿」是可行的。
